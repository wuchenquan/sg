# Slots 求职 Demo 需求文档

## 1. Demo 定位

项目名：**Demo Slots**

这是一个用于求职展示的互动 demo，目标不是复刻真钱老虎机，也不是做完整商业化产品，而是用一个可玩的轻量样机证明以下能力：

- 能拆解 Coin Master、Monopoly GO 类 T3-MCA slots 游戏的核心循环。
- 能把随机奖励、能量、建造、攻击、掠夺、护盾、倍率、活动和付费窗口组织成一个闭环。
- 能用清晰 UI 展示玩家体验、经济流和系统设计意图。
- 能用工程实现支撑设计，而不是只停留在文档层。

Demo 应在 3-5 分钟内让面试官看懂：

```text
一键随机输入
-> 高频奖励
-> 金币沉淀到成长目标
-> 攻击/掠夺制造情绪
-> 活动和排行榜制造短期目标
-> 能量耗尽触发继续游玩的付费窗口
```

## 2. 目标受众

主要面向：

- 系统策划 / 数值策划岗位。
- 休闲游戏 / 社交博彩 / F2P 游戏设计岗位。
- Gameplay programmer / prototype engineer 岗位。
- LiveOps、经济系统、留存转化相关岗位。

面试官应能从 demo 中看到：

- 你理解核心循环和商业化节奏。
- 你能把抽象设计转成可调参数和可运行系统。
- 你能处理玩家情绪、进度、资源回收、付费点之间的关系。
- 你知道哪些设计是展示用模拟，哪些不应直接变成真实诱导付费。

## 3. 设计边界

本 demo 使用虚拟货币和模拟付费意图，不接入真实支付，不展示真实价格，不引导真钱赌博。

所有“付费点”以设计分析形式呈现，例如：

- “此处是高转化窗口”
- “玩家资源耗尽且距离目标很近”
- “活动剩余时间制造压力”
- “倍率状态提高继续玩的动机”

Demo 要保持作品集友好：展示设计理解，而不是做一个掠夺式商业化产品。

## 4. 核心体验

玩家进入后直接看到可玩的主界面，不做营销式 landing page。

主体验由四个区域组成：

1. **Spin 区**
   - 一个醒目的主按钮：Spin / Roll。
   - 显示当前能量、金币、倍率、护盾。
   - 每次点击消耗能量，产出随机结果。

2. **进度区**
   - 展示一个小岛 / 城市 / 基地的 5 个升级槽。
   - 玩家用金币升级建筑。
   - 全部建筑完成后进入下一关，并解锁更高奖励参数。

3. **社交冲突区**
   - 展示 3-5 个异步玩家目标。
   - 随机触发攻击或掠夺。
   - 攻击会破坏目标进度或被护盾抵消。
   - 掠夺会偷取目标金币的一部分。
   - 被攻击 / 被掠夺事件会出现在事件流中。

4. **设计观察区**
   - 显示当前循环状态。
   - 显示本次 session 的关键指标。
   - 标注当前是否进入付费窗口。
   - 解释为什么此时玩家会想继续。

## 5. 核心循环需求

### 5.1 能量循环

初始能量：50。

每次 Spin 消耗：

- 基础消耗：1 能量。
- 若倍率为 x2/x5/x10，则消耗对应倍数能量。

能量不足时：

- Spin 按钮不可用或进入“补充能量”状态。
- 显示“模拟补充”按钮。
- 记录一次付费窗口事件。

Demo 中可提供一个“模拟补充 +50 能量”的按钮，但必须标注为 demo 行为。

### 5.2 随机结果

每次 Spin 从以下结果中抽取：

| 结果 | 作用 | 设计目的 |
| --- | --- | --- |
| Small Coins | 少量金币 | 高频正反馈 |
| Big Coins | 大量金币 | 奖励峰值 |
| Attack | 进入攻击小游戏 | 制造 PvP 情绪 |
| Raid | 进入掠夺小游戏 | 提供最大收益期待 |
| Shield | 获得防护 | 缓冲被攻击损失 |
| Combo | 连续奖励或倍率奖励 | 提高爽感和波动 |

默认命中率应偏高，让玩家几乎每次都有所得。

建议概率：

| 结果 | 初始概率 |
| --- | --- |
| Small Coins | 38% |
| Big Coins | 14% |
| Attack | 18% |
| Raid | 12% |
| Shield | 10% |
| Combo | 8% |

### 5.3 金币循环

金币来源：

- Spin 奖励。
- Attack 奖励。
- Raid 奖励。
- Combo 奖励。
- 活动奖励。

金币消耗：

- 建筑升级。
- 临时 boost。
- 进入下一关后的升级成本增长。

金币设计目标：

- 玩家要频繁感觉“快够了”。
- 金币不能无限堆积。
- 建造成本要推动玩家继续 Spin。
- Raid 只能偷取一部分金币，不能让目标崩盘。

### 5.4 建造循环

每一关有 5 个建筑槽：

- Gate
- Market
- Workshop
- Tower
- Shrine

每个建筑有 3 级。

升级消耗随关卡提升：

```text
cost = baseCost * buildingWeight * levelMultiplier * islandMultiplier
```

完成所有建筑后：

- 关卡 +1。
- 奖励表上调。
- 解锁一个短期活动奖励。
- 事件流显示“完成城市 / 岛屿”。

### 5.5 攻击系统

Attack 触发后进入轻量选择：

- 展示 3 个目标。
- 玩家选择一个目标。
- 系统展示攻击结果。

结果包括：

- 命中未保护建筑：目标建筑降 1 个损伤状态，玩家获得金币。
- 被护盾挡住：玩家仍获得少量安慰奖。
- 暴击：获得额外金币。

MCA 设计点：

- 玩家感觉自己选择了目标。
- 实际奖励和损失被系统控制在安全范围。
- 失败也有回报，避免负反馈过重。

### 5.6 掠夺系统

Raid 触发后进入三选一或五选三小游戏。

需求：

- 玩家选择若干格子。
- 每个格子揭示金币、空格、bonus。
- 最终获得一笔较大奖励。

设计目标：

- Raid 是 session 中最高期待事件。
- 结果波动大，但平均值受控。
- 被掠夺目标损失显示为“痛感事件”，但真实损失有上限。

### 5.7 护盾系统

Shield 用于抵挡一次攻击。

规则：

- 最多持有 3 个护盾。
- 超出上限时转化为金币或活动积分。
- 被攻击时自动消耗。

设计目的：

- 给玩家安全感。
- 让防守也能成为随机奖励的一部分。
- 降低被攻击带来的流失风险。

### 5.8 倍率系统

玩家可选择：

- x1
- x2
- x5
- x10

倍率影响：

- 能量消耗。
- 金币奖励。
- 活动积分。
- 攻击/掠夺收益。

倍率设计目标：

- 让高资源玩家加速消耗。
- 让玩家自选风险和节奏。
- 在活动剩余时间短时提高继续玩的冲动。

## 6. LiveOps 与付费窗口需求

### 6.1 Flash Tournament

短期排行榜活动，持续 5 分钟 demo 时间。

玩家通过 Spin、Attack、Raid 获得积分。

排行榜包含：

- 当前玩家。
- 8-12 个模拟玩家。
- 分数实时轻微变化。

触发付费窗口条件：

- 玩家排名第 2-4 名。
- 距离下一名差距小。
- 活动剩余时间低于 60 秒。
- 能量不足。

### 6.2 Reward Rush

短时加成活动。

效果：

- Coin 奖励 +50%。
- Raid 奖励 +80%。
- 持续 60-90 秒。

触发付费窗口条件：

- Reward Rush 生效。
- 玩家能量低。
- 建筑升级差一点完成。

### 6.3 Near Goal 付费窗口

当玩家满足以下条件时，标注一个“Near Goal Window”：

- 当前金币距离可升级建筑成本低于 20%。
- 能量低于 10。
- 活动仍在进行。

UI 应显示：

```text
差 420 coins 可升级 Tower
剩余 6 energy
Reward Rush 剩余 38s
```

### 6.4 Revenge 付费窗口

当玩家被模拟好友攻击后：

- 事件流显示来源玩家。
- 出现“Revenge”按钮。
- 若能量不足，标注“Revenge Window”。

设计目的：

- 展示社交情绪如何转化为继续游玩动机。

## 7. 数据与指标需求

Demo 需要实时显示以下指标：

| 指标 | 含义 |
| --- | --- |
| Spins | 本 session 点击次数 |
| Coins Earned | 累计获得金币 |
| Coins Spent | 累计消耗金币 |
| Energy Spent | 累计消耗能量 |
| Attacks | 攻击次数 |
| Raids | 掠夺次数 |
| Shields Used | 护盾消耗次数 |
| Upgrade Progress | 当前关卡完成度 |
| Paypoint Signals | 已触发付费窗口次数 |

指标不需要接入后端，本地状态即可。

## 8. UI / UX 需求

整体风格：

- 专业、可读、作品集友好。
- 不做赌场写实风，不做真钱赌博暗示。
- 使用清晰信息层级和轻量动效。
- 主按钮要有强反馈，但不能遮挡系统信息。

布局建议：

```text
顶部：资源栏 + 活动倒计时
左侧：主 Spin 区 + 随机结果
中间：城市/岛屿升级区
右侧：LiveOps + 指标 + 设计标注
底部：事件流
```

移动端可改为单列堆叠，但求职 demo 优先保证桌面观感。

必要状态：

- 初始状态。
- Spin 后奖励状态。
- Attack 选择状态。
- Raid 揭示状态。
- 能量不足状态。
- 建筑可升级状态。
- 付费窗口标注状态。
- 活动结束结算状态。

## 9. 技术需求

建议实现为本地可运行 Web demo。

优先技术栈：

- Vite
- TypeScript
- React
- CSS Modules 或普通 CSS

如安装依赖受限，可降级为：

- 单页 HTML
- 原生 TypeScript / JavaScript
- 纯 CSS

代码结构建议：

```text
src/
  data/
    rewards.ts
    liveops.ts
    opponents.ts
  simulation/
    rng.ts
    economy.ts
    attacks.ts
    raids.ts
    progression.ts
    paypoints.ts
  ui/
    components/
    panels/
  App.tsx
  main.tsx
```

核心逻辑必须从 UI 中拆出，方便面试时讲系统。

## 10. 非目标

本版本不做：

- 真实支付。
- 账号系统。
- 后端服务。
- 真实多人同步。
- 大规模美术资源。
- 复杂老虎机 reel 动画。
- 商店定价和真钱礼包。
- 长线内容生产工具。

## 11. 验收标准

MVP 完成后应满足：

- 打开页面即可开始体验。
- 玩家可连续 Spin，并看到资源变化。
- 至少包含 coins、attack、raid、shield、combo 六类结果。
- 可升级 5 个建筑并完成至少 1 个关卡循环。
- 能量不足时出现明确继续游玩窗口。
- 至少有 2 种 LiveOps 压力：排行榜和短时加成。
- 至少有 3 类付费窗口标注：Near Goal、Event Rush、Revenge。
- 指标面板能实时更新。
- 事件流能解释玩家刚刚发生了什么。
- 代码结构能支持面试讲解。

## 12. 面试展示脚本

推荐展示顺序：

1. 先展示主循环：Spin -> 得金币 -> 升级建筑。
2. 展示 Attack / Raid 如何把单机 slots 变成社交冲突。
3. 展示能量耗尽时，系统如何识别付费窗口。
4. 展示排行榜和短时活动如何改变玩家决策。
5. 打开代码讲 simulation 层如何与 UI 解耦。
6. 说明伦理边界：真实产品必须处理合规、未成年人保护和反沉迷。
